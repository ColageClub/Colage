## Architecture Overview

Colage is a **university social discovery platform** — think "see who's nearby on campus." The backend is a serverless AWS stack defined via SAM (Serverless Application Model):

- **Compute:** 20 Lambda functions (Node.js 20.x)
- **Auth:** AWS Cognito User Pool with custom OTP flow via SES
- **Database:** 6 DynamoDB tables (PAY_PER_REQUEST)
- **Storage:** S3 bucket for profile photos (presigned URLs, no CloudFront yet)
- **Real-time:** WebSocket API Gateway for live location broadcasting
- **Billing:** Nightly cron Lambda for ad billing via Stripe
- **API:** SAM HttpApi (API Gateway v2) with Cognito JWT authorizer

### High-Level Flow

```
iOS App → HTTP API (Cognito JWT auth) → Lambda → DynamoDB
                                                → S3 (photos)
                                                → Cognito (auth ops)
        → WebSocket API → Lambda → DynamoDB (locations + connections)
                                 → Broadcast to peers via API GW Management API
```

## Auth Flow — End to End

### Signup Flow

```
1. POST /auth/email/verify { email: "user@umich.edu" }
   → Rate limit check (5/min per IP, 3/5min per email)
   → Check if user exists in Cognito (AdminGetUser)
   → Generate 6-digit OTP, store in OTP table (10min TTL)
   → If new user: create Cognito user (SignUp with random temp password)
   → Send OTP via SES email
   → Returns { message, existing: bool }

2. POST /auth/email/confirm { email, code }
   → Rate limit (10/min per IP, 5/5min per email)
   → Look up OTP in DynamoDB
   → Verify not expired, code matches
   → AdminConfirmSignUp in Cognito
   → Delete OTP record
   → Returns { verified: true, universityDomain }

3. POST /users { email, displayName, universityDomain, ... }
   → No auth required (!)
   → Rate limit (3/hour per IP)
   → Validates .edu email, creates profile in UsersTable
   → Auto-creates or increments university in UniversitiesTable
   → Returns { profile }
```

### Login Flow

```
4. POST /auth/login { email, deviceId }
   → Generates deterministic password from email: `Colage!${base64(email).slice(0,20)}1`
   → AdminSetUserPassword (sets the deterministic password)
   → AdminInitiateAuth with ADMIN_USER_PASSWORD_AUTH
   → Attempts to store deviceId in UsersTable (wrong key schema — see bugs)
   → Returns { accessToken, idToken, refreshToken, expiresIn }
```

### Token Refresh

```
5. POST /auth/refresh { refreshToken }
   → AdminInitiateAuth with REFRESH_TOKEN_AUTH
   → Returns { accessToken, idToken, expiresIn }
```

### Phone Verification (Placeholder)

```
POST /auth/phone/verify { phone, email }  → Always returns success (no Twilio)
POST /auth/phone/confirm { phone, code }  → Accepts any 6-digit code
```

## Function-by-Function Analysis

### `auth-email-verify`

**Purpose:** Initiate .edu email verification by sending an OTP code.

**Flow:**
1. Rate limit by IP (5/min) and email (3/5min)
2. Validate `.edu` email format
3. Check if user exists in Cognito (`AdminGetUser`)
4. Generate 6-digit OTP via `Math.random()`
5. Store OTP in DynamoDB with 10-min TTL
6. If new user: `SignUp` in Cognito with random temp password
7. Send OTP via SES (HTML + plain text email)

**Input Validation:** ✅ Good — validates email format, .edu suffix, rate limits both IP and email.

**Security Concerns:**
- 🟡 **MEDIUM:** OTP generated with `Math.random()` — not cryptographically secure. Should use `crypto.randomInt()`.
- ✅ Rate limiting is properly layered (IP + email)
- ✅ Temp password is random and never exposed

**DynamoDB Ops:** `PutCommand` → OTPTable  
**Cognito Ops:** `AdminGetUserCommand`, `SignUpCommand`  
**SES Ops:** `SendEmailCommand`

**Error Handling:** Good — catches `UserNotFoundException` specifically, wraps unknown errors.

---

### `auth-email-confirm`

**Purpose:** Verify the OTP code and confirm the Cognito user.

**Flow:**
1. Rate limit by IP (10/min) and email (5/5min)
2. Validate email and OTP format
3. Fetch OTP record from DynamoDB
4. Check expiry, compare code
5. `AdminConfirmSignUp` in Cognito (ignores `NotAuthorizedException` for already-confirmed)
6. Delete OTP record
7. Extract root `.edu` domain and return

**Input Validation:** ✅ Good — validates email, OTP format, checks expiry.

**Security Concerns:**
- 🟢 OTP comparison is simple string equality — no timing attack concern since it's a 6-digit number (low entropy)
- ✅ OTP is deleted after use
- ✅ Expired OTPs are cleaned up

**DynamoDB Ops:** `GetCommand` + `DeleteCommand` → OTPTable  
**Cognito Ops:** `AdminConfirmSignUpCommand`

**Edge Case:** Domain extraction collapses subdomains: `user@cs.umich.edu` → `umich.edu`. This is intentional (groups sub-departments under parent university).

### `auth-login`

**Purpose:** Passwordless login — generates deterministic Cognito password from email, authenticates, returns tokens.

**Flow:**
1. Parse email and deviceId from body
2. Generate deterministic password: `Colage!${Buffer.from(email).toString('base64').slice(0,20)}1`
3. `AdminSetUserPassword` — sets this password permanently
4. `AdminInitiateAuth` with `ADMIN_USER_PASSWORD_AUTH`
5. Attempt to store deviceId in UsersTable (broken — see bugs)
6. Return tokens

**Input Validation:** 🟡 Minimal — checks email contains `@` and deviceId exists, but no `.edu` validation, no sanitization, no rate limiting.

**Security Concerns:**
- 🔴 **CRITICAL:** Deterministic password derivation means anyone who knows an email can compute the password. The "security" relies entirely on Cognito being the only caller of `AdminSetUserPassword`, but the password itself is guessable.
- 🔴 **CRITICAL:** No rate limiting on login endpoint. Brute-force attacks possible.
- 🟠 **HIGH:** Does not use shared `response()` helper — returns raw `response()` local function without `X-Content-Type-Options` header.
- 🟡 **MEDIUM:** deviceId storage uses `email` as DynamoDB key, but UsersTable PK is `userId`. The UpdateCommand will either fail silently or create a phantom item.

**DynamoDB Ops:** `UpdateCommand` → UsersTable (with wrong key — `email` instead of `userId`)  
**Cognito Ops:** `AdminSetUserPasswordCommand`, `AdminInitiateAuthCommand`

**Error Handling:** Catches `UserNotFoundException` specifically, generic 500 for others. Missing: `NotAuthorizedException`, `UserNotConfirmedException`.

### `create-profile`

**Purpose:** Create a new user profile after email verification.

**Flow:**
1. Rate limit (3/hour per IP)
2. Validate required fields: `email`, `displayName`, `universityDomain`
3. Validate `.edu` email and domain
4. Sanitize social links (whitelist of platforms, max 10)
5. Generate UUID for `userId`
6. `PutCommand` with `ConditionExpression: 'attribute_not_exists(userId)'` (prevents overwrites)
7. Auto-create or increment university in UniversitiesTable

**Input Validation:** ✅ Good — validates email format, domain, display name length, sanitizes all text fields, whitelists social platforms.

**Security Concerns:**
- 🔴 **CRITICAL:** **No auth required** (`Auth: Authorizer: NONE`). Anyone can create a profile with any .edu email. There's no check that the email was actually verified via the OTP flow.
- 🟡 **MEDIUM:** No duplicate email check. The `ConditionExpression` only prevents duplicate `userId` (which is random UUID). Two profiles could exist for the same email.

**DynamoDB Ops:** `PutCommand` → UsersTable, `GetCommand` + `UpdateCommand`/`PutCommand` → UniversitiesTable

**Edge Case:** University auto-creation generates name as `domain.replace('.edu', '').toUpperCase()` — e.g., `umich` → `"UMICH University"`. Functional but crude.

2. Fetch user from UsersTable (to get email + universityDomain)
3. Delete from UsersTable
4. Delete from LocationsTable

**Input Validation:** Minimal — only checks `userId` exists in path.

- 🟡 **MEDIUM:** Does not decrement university `memberCount`.
- 🟡 **MEDIUM:** Does not delete from ConnectionsTable or clean up S3 photos.

**DynamoDB Ops:** `GetCommand` + `DeleteCommand` → UsersTable, `DeleteCommand` → LocationsTable  
**Cognito Ops:** `AdminDeleteUserCommand`

1. Get `email` from query string parameter
2. **Scan** UsersTable with filter `email = :email`
3. Return profile (strips `email` from response)

**Input Validation:** Checks email query param exists.

**Input Validation:** Only checks domain exists. No `.edu` validation.

**DynamoDB Ops:** `GetCommand` + conditional `PutCommand` → UniversitiesTable

**Known universities hardcoded:** `umich.edu`, `harvard.edu`, `stanford.edu`.

### `get-nearby`

**Purpose:** Find students near a location at a given university, using geohash spatial indexing.

**Flow:**
1. Rate limit (30/min per IP)
2. Validate `domain` (.edu), optional `lat`/`lng`, optional `maxDistance`
3. If coordinates provided:
   - Compute geohash precision based on search radius
   - Get center cell + 8 neighbors
   - Query `geohash-index` GSI for each cell (9 parallel queries)
   - Deduplicate by userId
4. If no coordinates: query all locations for the university
5. Batch get profiles from UsersTable (100 at a time)
6. Filter out `isVisible: false`
7. Calculate haversine distance, filter by maxDistance, sort

**Input Validation:** ✅ Good — validates domain, coordinates, caps maxDistance at 10,000 feet.

**Security:** ✅ Requires Cognito JWT. Rate limited.

**DynamoDB Ops:** Up to 9 `QueryCommand` → LocationsTable (geohash-index GSI), `BatchGetCommand` → UsersTable

**Performance:** Well-designed. Geohash approach avoids full table scans. 500-student cap implicit from TTL (5-min location expiry limits active results).

**Edge Case:** The "no coordinates" fallback queries ALL locations for a university — could return a lot of data for large schools.

### `ws-location-update`

**Purpose:** Receive a location update, store it, and broadcast to same-university peers.

**Flow:**
1. Parse JSON body, extract `data` object
2. Validate `userId` (string, max 128 chars), coordinates
3. Clamp floor (-2 to 200), default altitude to 0
4. Look up connection to get university domain
5. Compute geohash (precision 7)
6. Store location in LocationsTable with 5-min TTL
7. Fetch minimal profile (displayName, photo, major) for broadcast payload
8. Query all connections for the same university
9. Broadcast location update to up to 500 peers
10. Clean up stale connections (410 Gone) during broadcast

**Input Validation:** ✅ Good — validates userId type/length, coordinates, clamps floor range.

**Security Concerns:**
- 🔴 **CRITICAL:** **No auth on the WebSocket** (inherited from `ws-connect`). Anyone connected can send location updates for any userId.
- 🟡 **MEDIUM:** The `userId` in the location update message is trusted — not verified against the connection's stored userId. A connected user could broadcast locations for other users.

**DynamoDB Ops:** `GetCommand` → ConnectionsTable, `PutCommand` → LocationsTable, `GetCommand` → UsersTable, `QueryCommand` → ConnectionsTable (by-university GSI), potential `DeleteCommand` → ConnectionsTable (stale cleanup)

**Performance:** Capped at 500 peer broadcasts. Uses `Promise.all` for parallel sends. Good design.

### `billing-nightly`

**Purpose:** Nightly cron job to bill businesses for ad impressions.

**Flow:**
1. Scan all active ads
2. Count advertisers per school (for dynamic CPM)
3. Group ads by business
4. For each business:
   a. Get business record
   b. For each ad: get daily spend, calculate `impressions × CPM / 1000`, cap at `dailyBudget`
   c. Update ad's `totalSpend`
   d. Record actual charge in daily_spend
   e. Charge total via Stripe customer balance transaction
   f. Update cached balance in DynamoDB
   g. If balance ≤ $0: pause all ads
   h. If balance < $10: log warning (email TODO)

**CPM Tiers:**
| Advertisers at School | CPM |
|---|---|
| ≤10 | $2 |
| ≤30 | $3 |
| ≤60 | $4 |
| ≤100 | $5 |
| >100 | $6 |

**Security Concerns:**
- 🔴 **CRITICAL:** `STRIPE_SECRET_KEY` is hardcoded as `"CONFIGURE_ME"` in the SAM template. If deployed as-is, Stripe calls would fail. The TODO says "Move to SSM Parameter Store."
- 🟡 **MEDIUM:** Uses `ScanCommand` for active ads — will get slow with many ads. Should use a GSI on status.

**DynamoDB Ops:** `ScanCommand` → AdsTable, `GetCommand` → BusinessesTable, `GetCommand` → DailySpendTable, `UpdateCommand` → AdsTable, `UpdateCommand` → DailySpendTable, `UpdateCommand` → BusinessesTable

**Error Handling:** ✅ Good — per-business error isolation, detailed logging, result summary.

**Edge Cases:**
- Bills for `yesterdayString()` — correct for a midnight cron.
- Race condition: if the function runs multiple times, charges could double. No idempotency key.

## WebSocket Lifecycle

```
1. Client connects: wss://...?domain=umich.edu&userId=abc-123
   → ws-connect stores { connectionId, domain, userId } in ConnectionsTable (24h TTL)

2. Client sends: { "action": "location.update", "data": { userId, latitude, longitude, ... } }
   → ws-location-update:
     - Stores location in LocationsTable (5min TTL)
     - Fetches sender's profile for broadcast payload
     - Queries all connections for same university
     - Broadcasts to up to 500 peers (excluding sender)
     - Cleans up stale connections (410 Gone)

3. Client disconnects:
   → ws-disconnect deletes connection from ConnectionsTable

4. Stale cleanup:
   - ConnectionsTable has 24h TTL (DynamoDB auto-delete)
   - LocationsTable has 5min TTL (locations auto-expire)
   - 410 responses during broadcast trigger immediate connection cleanup
```

**Critical Gap:** No authentication on WebSocket. The `userId` is self-reported via query string and message body. Any client can impersonate any user.

## Bugs & Issues

### Confirmed Bugs

1. **🔴 `auth-login` device storage uses wrong key** — The `UpdateCommand` uses `Key: { email: normalizedEmail }` but UsersTable's primary key is `userId`. This will either:
   - Create a phantom item with `email` as a hash key attribute (DynamoDB allows this if the attribute name matches)
   - Actually: since UsersTable is defined with `userId` as the HASH key, this `UpdateCommand` will create a NEW item with the `email` value in the `userId` field. This is a data corruption bug.

2. **🟠 `get-me` uses Scan instead of GSI** — The `by-email` GSI exists on UsersTable but `get-me` does a full table Scan with a FilterExpression. Should use `QueryCommand` on the `by-email` index.

3. **🟡 `billing-nightly` SCans for active ads** — No GSI on ad status. Will degrade as ad volume grows.

### Potential Race Conditions

1. **University memberCount** — `create-profile` does a `GetCommand` then conditional `UpdateCommand`/`PutCommand`. Two simultaneous profile creations for the same university could both try to `PutCommand` a new university (one will fail silently).

2. **Billing double-charge** — No idempotency key. If the Lambda retries or runs twice, charges double.

3. **OTP race** — Two rapid verify requests for the same email could generate two OTPs. The second overwrites the first (PutCommand, no condition). Not a bug per se, but the user's first OTP code won't work.
